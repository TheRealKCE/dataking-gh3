## Potential conflicts and manual review items

- Multiple migrations intentionally redefine the same RPCs, policies, triggers, and RLS posture over time; the master script orders functions/triggers/policies after base tables and uses repeat-safe guards/replacements.
- PostgreSQL cannot use native `IF NOT EXISTS` for every object class; enum/type creation, constraints, and policies are wrapped in catalog checks or duplicate-object handlers.
- Static SQL parsing cannot prove semantic equivalence for repeated table definitions or all function bodies. Test in a staging Supabase project before production use.
- Inserts were made repeat-safe with `ON CONFLICT DO NOTHING` where a statement had no conflict clause. Review seed rows that should update existing values instead of being skipped.
- Trigger definitions are emitted as `DROP TRIGGER IF EXISTS ...; CREATE TRIGGER ...` so reruns use the latest trigger body.
- `SET check_function_bodies = off` is used so functions can be created before every referenced relation is fully available; this is appropriate for consolidated deployment but warrants staging validation.

## Final validation summary

- Dependency ordering was normalized into extensions, tables, column backfills, table alters, constraints, indexes, functions, triggers, views, policies, grants, seed data, and remaining operational blocks.
- Duplicate object definitions were detected and listed above. Superseded duplicate named constraints are skipped in the master script so later migrations win.
- Circular dependencies were not detected by the static table-reference sorter; unresolved procedural dependencies remain flagged for staging validation because RPC bodies may reference runtime-only objects.
- Existing-object failure modes were mitigated with `IF NOT EXISTS`, `CREATE OR REPLACE`, policy catalog checks, trigger drops before recreate, guarded constraint creation, and conflict-safe seed inserts.
